### T2 — Tree layout engine (pure function)

**Description:** Implement a custom recursive tree layout algorithm as a pure function. No dagre/elkjs dependency — BT is a strict hierarchy, not a general DAG. The function computes new positions for all nodes given a `BehaviorTree`, snapping to the grid.

**Acceptance criteria:**
- [x] New file `src/core/layout/tree-layout.ts` exports `computeTreeLayout(tree, options): Map<string, {x, y}>` with `LayoutOptions` interface
- [x] Root centered over its children; children below, siblings left-to-right by connection `order` value
- [x] All returned positions are snapped to `gridSize`
- [x] Orphaned nodes (no parent, not Root) placed in a row directly above Root, starting at Root's column. **Refined during T4 manual test** — original "column to the right of main tree" placed orphans 10000 px from Root for wide trees (per `valid_tree_04.json` regression).
- [x] **Root anchored to user position** (refined during T4 manual test). The function translates the whole layout so Root ends up at its existing `node.position`, instead of letting the recursive algorithm place Root at the leaf-spread midpoint. Prevents jarring "tree teleports across the canvas" when the user clicks Layout.
- [x] Connection `order` values are NOT modified — covered by input-immutability test in T5
- [x] Function is pure: no store access, no side effects
- [x] ~75 lines implementation

**Verification:**
- [x] `npm run typecheck` green
- [x] 100% coverage on `tree-layout.ts` from T5

**Dependencies:** None

**Files likely touched:**
- `src/core/layout/tree-layout.ts` (new)

**Estimated scope:** M

### T4 — Auto layout button + store integration

**Description:** Add a "Layout" button to the toolbar that triggers the tree layout engine and applies the computed positions to all nodes as a single undo step. After applying, recenter the viewport on Root.

**Acceptance criteria:**
- [x] "Layout" button in toolbar — icon (custom inline SVG, tree glyph) + "Layout" text, placed next to the Grid toggle
- [x] Clicking the button:
  1. Calls `computeTreeLayout(tree, LAYOUT_OPTIONS)` with `gridSize=GRID_SIZE`, `nodeWidth=NODE_WIDTH`, `nodeHeight=NODE_HEIGHT`, `gapX=50`, `gapY=50` (constants from `src/core/config/grid.ts`)
  2. Applies the resulting `Map<string, {x, y}>` to all node positions via new store action `applyLayout`
  3. `applyLayout` uses `withHistory` so the entire re-positioning is a single undo step
  4. Calls `setCenter(rootX + NODE_WIDTH/2, rootY + NODE_HEIGHT/2, { zoom: getZoom(), duration: 300 })` on the React Flow instance — pans to Root's visual midpoint while preserving the user's current zoom. **`zoom` must be passed explicitly** — `setCenter` defaults to 1.0 otherwise (see `memory/reference_xyflow_setcenter.md`).
- [x] Connection `order` values are NOT changed
- [x] Works correctly with orphaned nodes (placed in row above Root)
- [x] Works with empty tree (single Root) — `applyLayout` returns `{}` when no positions actually change (no-op, no history snapshot)
- [x] **`<ReactFlowProvider>` lifted from `Canvas.tsx` to `App.tsx`** so `useReactFlow()` works in toolbar-mounted hooks. `Canvas` and `CanvasInner` merged into a single `Canvas` export.

**Verification:**
- [x] Manual browser test 2026-04-26 with a real robot BT (~90 nodes + intentional orphan): clean top-down layout, Root anchored, orphan above Root, viewport centers on Root, zoom preserved at any level (tested at 50% / 100% / 150%), undo restores in one step, redo re-applies.
- [x] All e2e + unit tests pass after lifting the provider — Toolbar tests now wrap renders in `<ReactFlowProvider>` via a local helper.

**Dependencies:** T2 (needs `computeTreeLayout`)

**Files touched:**
- `src/store/bt-store.ts` — `applyLayout` action
- `src/hooks/useApplyLayout.ts` — new hook bridging store + React Flow camera
- `src/components/toolbar/Toolbar.tsx` — Layout button + `LayoutIcon`
- `src/App.tsx` — wrap editor branch in `<ReactFlowProvider>`
