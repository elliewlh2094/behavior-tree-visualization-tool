### T1 — Preferences store with defaults

**Description:** Create a new Zustand store separate from `bt-store.ts` that holds all user-customizable preferences. This store has a different lifecycle: persisted to localStorage, never in undo/redo, survives tree open/close. Initially created without persistence (added in T3).

**Acceptance criteria:**
- [x] New file `src/store/preferences-store.ts` exports `usePreferencesStore`
- [x] State shape:
  ```ts
  interface Preferences {
    canvasBg: string;          // default: '#f8fafc' (slate-50)
    gridLineColor: string;     // default: current ReactFlow grid color
    edgeColor: string;         // default: current edge color
    edgeThickness: number;     // default: 1, range 1–2.5
    nodeBorderThickness: number; // default: 1, range 1–2
    nodeBgByKind: Record<string, string>; // defaults from kind-visuals.ts
    theme: 'light' | 'dark' | 'system'; // default: 'light'
  }
  ```
- [x] `setPreference(key, value)` action — updates a single preference
- [x] `resetAll()` action — restores all defaults
- [x] Defaults are exported as a named constant for reference by other modules
- [x] Store is NOT connected to bt-store undo/redo

## Mid-flight changes (deviations from the original plan)

Two scope changes landed between Phase 2 and Phase 3 in response to user
feedback during browser review. Both are documented commits on the v1.3
branch and are not covered by T1–T11 above.

### Phase 2.5 — Preferences model redesign (`f1a4883`)

**What changed:** The original Phase 2 (`c70cc32`) shipped a Settings panel
where users could customize *every* color and thickness on the chrome —
canvas bg, grid color, edge color, edge thickness, border thickness, plus
per-kind node bg as free hex. After visual review the user asked to
narrow the customizable surface to **node colors only** (one Tailwind hue
family per `NodeKind`), with the chrome fully designer-owned.

**Why:** Cleaner ownership split — chrome is product identity (designer's
job), node colors are the user's *semantic* palette (which node type is
which color in *their* tree). Picking a single family per kind also
encodes an entire color quadruple (light/dark × bg/border) rather than
single hexes that don't adapt across themes.

**What landed:**
- New `src/components/canvas/color-families.ts` — 22 Tailwind hue families
  × 10 shades, role→shade tables for light/dark, `nodeVar(role, kind)` helper.
- Preferences store narrowed to `{ nodeFamilyByKind, theme, showGrid }`.
- Persist version bumped 1 → 2 with migrate dropping v1 customizations.
- New `useResolvedTheme` hook centralizes light/dark resolution.
- `usePreferencesSync` writes 6 vars × 8 kinds = 48 CSS custom properties.
- `ColorPicker` rewritten as a click-to-expand family picker.
- `SettingsPanel` collapsed to Nodes + Theme + Reset (Canvas/Edges sections gone).

### Phase 2.6 — UI refresh (`6a846f3`)

**What changed:** Five user-driven design tweaks bundled because they
overlap on the same files:

1. Settings becomes a peer **tab** in the right Sidebar alongside
   Properties (no more gear button, no more slide-out drawer).
2. Grid Background **moves into the Settings tab** below Theme as a
   2-segment On/Off pill. Old iOS-style switch removed from toolbar.
3. **Chrome tokens** introduced (`--bt-panel-bg` slate-50, `--bt-border`
   slate-300 matching toolbar separator hairlines, `--bt-tab-inactive-bg`
   slate-300, `--bt-text-primary`/`--bt-text-secondary`).
4. **Toolbar buttons** gain icons in the LayoutIcon style: Undo, Redo,
   Validate, Open, Save (14×14, currentColor, strokeWidth 1.5).
5. **Node shade tweaks** per `docs/v1.3-color-reference.md` after "too
   pale / too dark" feedback — Light bg `-50→-100`, Dark whole palette
   shifted up one Tailwind step (`-900→-800`, etc).

### Ship — WCAG AA contrast fixes (`681bbf9`)

Lighthouse audits in dark mode flagged three contrast failures:
- StartScreen "New Tree" button: removed `dark:bg-sky-600` override
  (sky-700 works in both modes at 6.4:1 contrast).
- `--bt-text-secondary` dark value bumped slate-400 → slate-300.
- `--bt-tab-inactive-bg` dark value moved slate-700 → slate-900
  (also restores the light-mode parallel: inactive tab darker than
  panel body).
- Plus `body { @apply bg-white dark:bg-slate-900 }` so axe-core has a
  correct fallback when walking up from var-driven inline backgrounds.

Final scores all four audits, both modes: **Perf 97 / A11y 100 / BP 100 /
SEO 83** (StartScreen) and **Perf 0 / A11y 100 / BP 100 / SEO 67**
(Editor — Perf 0 matches v1.2 baseline). Reports archived to
`docs/lighthouse/v1.3-{audit,editor-audit}-{light,dark}.report.{html,json}`.
