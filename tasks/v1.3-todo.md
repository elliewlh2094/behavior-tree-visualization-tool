## Phase 1: Infrastructure

### T1 — Preferences store with defaults

**Description:** Create a new Zustand store separate from `bt-store.ts` that holds all user-customizable preferences. This store has a different lifecycle: persisted to localStorage, never in undo/redo, survives tree open/close. Initially created without persistence (added in T3).

**Acceptance criteria:**
- [ ] New file `src/store/preferences-store.ts` exports `usePreferencesStore`
- [ ] State shape:
  ```ts
  interface Preferences {
    canvasBg: string;          // default: '#f8fafc' (slate-50)
    gridLineColor: string;     // default: current ReactFlow grid color
    edgeColor: string;         // default: current edge color
    edgeThickness: number;     // default: 1, range 1–2.5
    nodeBorderThickness: number; // default: 1, range 1–2
    nodeBgByKind: Record<string, string>; // defaults from kind-visuals.ts
    theme: 'light' | 'dark' | 'system'; // default: 'light'
  }
  ```
- [ ] `setPreference(key, value)` action — updates a single preference
- [ ] `resetAll()` action — restores all defaults
- [ ] Defaults are exported as a named constant for reference by other modules
- [ ] Store is NOT connected to bt-store undo/redo

**Verification:**
- [ ] `npm run typecheck` green
- [ ] Unit test at `tests/unit/store/preferences-store.test.ts`:
  - Initial values match defaults
  - `setPreference` updates individual values
  - `resetAll` restores all defaults

**Dependencies:** None

**Files likely touched:**
- `src/store/preferences-store.ts` (new)
- `tests/unit/store/preferences-store.test.ts` (new)

**Estimated scope:** S

### T2 — CSS custom properties migration

**Description:** Replace all hardcoded color values in components with CSS custom properties driven by the preferences store. This is the bridge between the store and the visual output — when a preference changes, the CSS variable updates and all consumers react without re-rendering.

**Acceptance criteria:**
- [ ] `src/styles/tailwind.css` declares CSS custom properties in `:root` with default values:
  - `--bt-canvas-bg`, `--bt-grid-color`, `--bt-edge-color`, `--bt-edge-thickness`, `--bt-node-border-thickness`
  - `--bt-node-bg-root`, `--bt-node-bg-sequence`, etc. (one per kind)
- [ ] A `PreferencesProvider` component (or effect in `App.tsx`) syncs the preferences store to `document.documentElement.style.setProperty()` on every change
- [ ] `Canvas.tsx` reads `--bt-canvas-bg` and `--bt-grid-color` instead of hardcoded values
- [ ] `BTNode.tsx` reads `--bt-node-bg-*` and `--bt-node-border-thickness` instead of hardcoded values
- [ ] `kind-visuals.ts` updated to provide CSS variable names alongside static defaults
- [ ] Edge styling reads `--bt-edge-color` and `--bt-edge-thickness`
- [ ] No visual change with default preferences (backward compatible)

**Verification:**
- [ ] `npm run typecheck` green
- [ ] Manual browser test: app looks identical with defaults
- [ ] DevTools: CSS variables are present on `:root` and consumed by components
- [ ] Changing a preference in store → visual updates immediately (testable via DevTools console)

**Dependencies:** T1 (needs preferences store for variable values)

**Files likely touched:**
- `src/styles/tailwind.css`
- `src/App.tsx` (or new `PreferencesProvider` wrapper)
- `src/components/canvas/Canvas.tsx`
- `src/components/canvas/BTNode.tsx`
- `src/components/canvas/kind-visuals.ts`

**Estimated scope:** M

### T3 — Zustand persist middleware for preferences

**Description:** Wrap the preferences store with Zustand's `persist` middleware to automatically save/load from localStorage. Handles versioning and corruption gracefully.

**Acceptance criteria:**
- [ ] `preferences-store.ts` wrapped with `persist` middleware
- [ ] Storage key: `bt-visualizer-preferences`

- [ ] Set garbage in `bt-visualizer-preferences` → refresh → defaults applied, no errors

  - Text colors (headings, body, muted text)
  - Border colors
  - Input/select backgrounds and borders
  - Button hover/active states
  - Shadows (darker shadows in dark mode)
- [ ] Components to update:

### T9 — FOUC prevention script

**Description:** Add an inline script to `index.html` that applies the saved theme before the app renders, preventing a flash of the wrong theme on page load.

**Acceptance criteria:**
- [ ] Inline `<script>` in `<head>` of `index.html` (before any CSS/JS loads)
- [ ] Script reads `bt-visualizer-preferences` from localStorage
- [ ] If saved theme is "dark", adds `dark` class to `<html>` immediately
- [ ] If saved theme is "system", checks `matchMedia('(prefers-color-scheme: dark)')` and adds class if needed
- [ ] If no saved preference or corrupt data, does nothing (defaults to light)
- [ ] Script is <20 lines, synchronous, no external dependencies

**Verification:**
- [ ] Manual browser test: set dark mode → refresh → no flash of light mode
- [ ] Manual browser test: set system mode + OS dark → refresh → no flash
- [ ] Manual browser test: clear localStorage → refresh → light mode, no errors

**Dependencies:** T6 (needs the theme logic to be defined so the script matches the runtime behavior)

**Files likely touched:**
- `index.html`

**Estimated scope:** XS
