# v1.3 Color Reference

A snapshot of every color value currently used in the UI as of `f1a4883`
(v1.3 Phase 2.5). Use this as a working sheet for tuning shades; when a
value is changed, update the cell here AND its source-of-truth file in
the same commit.

## How the color system is layered

There are three layers, from outside in:

1. **Designer-owned chrome tokens** тАФ declared in `src/styles/tailwind.css`
   `:root { тАж }`. Light values land here today; `.dark { тАж }` overrides
   land in v1.3 Phase 3 (T8). Users cannot change these.
2. **User-owned node tokens (per-role ├Ч per-kind)** тАФ also declared in
   `tailwind.css` `:root` for first-paint correctness, but the *runtime*
   values come from `usePreferencesSync` writing to
   `document.documentElement.style.setProperty(...)`. Each kind picks a
   *family* (Tailwind hue) and the sync hook resolves which shade to use
   for each role based on `useResolvedTheme()`.
3. **Hardcoded one-offs** тАФ values that escape the token system. Mostly
   selection-state edges and a temporary hardcode for the React Flow
   `<Background color>` prop. Listed at the bottom for visibility; goal
   is to reduce these to zero.

## Suggested adjustments for the "too light/too dark" feedback

Below are starting points, not commitments тАФ review and tweak.

### Light mode

The complaint is "too pale". The cleanest single-knob fix is bumping
the bg shade from `-50` to `-100` while leaving the rest alone. That
keeps the soft-pastel character but adds enough chroma that nodes read
clearly even against a chrome that's also slate-50 (per Item 3).

```ts
// color-families.ts ROLE_SHADE_LIGHT
{ bg: 100, border: 300, accent: 700, accentBg: 200, ring: 300, borderSelected: 600 }
//      тЖС                              тЖС              тЖС
//   was 50                         was 100        was 200
```

`accentBg` and `ring` shift up one step too so they keep their
relationship to `bg` (accent pill stays ~one shade darker than node bg;
ring stays ~one shade darker than border).

### Dark mode

The complaint is "too dark". Bumping bg from `-900` to `-800` (or even
`-700` for high-saturation families) opens up readability without
losing the dark feel. Borders and accents need to lighten in step.

```ts
// color-families.ts ROLE_SHADE_DARK
{ bg: 800, border: 500, accent: 200, accentBg: 700, ring: 600, borderSelected: 200 }
//      тЖС       тЖС             тЖС              тЖС           тЖС              тЖС
//   was 900  was 400      was 300         was 800     was 700        was 300
```

This shifts the whole dark-mode palette up one Tailwind step. Net
effect: node bgs are visibly tinted (not near-black), borders are
saturated and clear, accent text reads as a bright label.

### Trade-off to consider before applying

Both adjustments narrow the contrast range between roles (since the
shade-index gap from bg тЖТ border тЖТ accent is the same number of steps
but those steps cover less perceptual range as you move toward the
middle of the scale). On low-saturation families (Slate, Gray, Stone,
Neutral, Zinc), this can make bg and border look more similar than
intended. If that turns out to be a problem after testing on Slate
nodes specifically, we can split the role table into two: one for
neutral families (keep the wider gaps) and one for saturated families
(use the bumped values).

## How to test a shade change

1. Edit `ROLE_SHADE_LIGHT` / `ROLE_SHADE_DARK` in `color-families.ts`.
2. `npm run dev` тАФ open the editor, drop one of every node kind to compare.
3. Toggle Theme тЖТ Dark in Settings to compare both modes.
4. Run `npm run typecheck && npx vitest run && npx playwright test e2e/canvas-controls.spec.ts`.
5. Update the "Resolved hex per (kind, role)" tables in this doc.

If you change anything in the token system itself (add/remove a role,
rename a CSS var), also update:
- `src/components/canvas/color-families.ts` (`NodeRole`, `nodeVar`, role tables)
- `src/styles/tailwind.css` `:root` defaults
- `src/hooks/usePreferencesSync.ts` (loop over roles)
- `tests/unit/store/preferences-store.test.ts` (assertions)
