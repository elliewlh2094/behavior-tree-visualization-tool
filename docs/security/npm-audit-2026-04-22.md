# npm audit — investigation record

> Date: 2026-04-22
> Context: post-S0.5 scaffold (commit `7317242`). First `npm audit` after initial `npm install`.
> Status: **Investigation closed. Decision: defer remediation; revisit at S10 and S11.**
> Next revisit: S10 (PWA wiring, when `vite-plugin-pwa` major upgrade happens anyway) and S11 (pre-launch audit).

## 1. Executive summary

`npm install` after the S0.5 scaffold produced **10 audit findings (6 moderate, 4 high)**. Investigation showed:

- **All 10 are in devDependencies.** The production bundle (`dist/`) contains none of the vulnerable code.
- **Only 3 distinct CVEs.** The other 7 entries are the same three CVEs surfacing through transitive dependency chains (npm audit reports each affected package in the chain separately).
- **No practical exposure for deployed users.** The vulnerabilities affect the developer's dev server, the build process, and the test runner. Nothing reaches an end user of the PWA.
- **Developer exposure is near-zero.** The one theoretically-exploitable-in-dev CVE (`esbuild` dev server) requires an adversarial-website-while-dev-server-is-running scenario with port guessing. Not meaningful for a local-only dev loop.

**Decision: defer remediation** rather than run `npm audit fix --force` now. Rationale: the proposed automatic fix would bump Vite 5 → 7/8, Vitest 2 → 4, and vite-plugin-pwa 0.20 → 1.2 in one shot, breaking the freshly-verified green baseline for a threat model that doesn't reach users. Planned follow-up in §6.

## 3. The three distinct CVEs

The audit surfaced 10 entries. Each is explained below by root CVE, with the transitive packages it manifests in.

### 3.1 CVE — esbuild dev server request leak

| Field | Value |
|---|---|
| Advisory | [GHSA-67mh-4wv8-2f99](https://github.com/advisories/GHSA-67mh-4wv8-2f99) |
| CWE | CWE-346 (Origin Validation Error) |
| CVSS | 5.3 (moderate) |
| Vulnerable range | `esbuild ≤ 0.24.2` |
| Fix | Vite 7+ (which bundles esbuild 0.25+) |

**What it does.** A website the developer visits can initiate cross-origin requests to the local esbuild dev server and read the response, because esbuild does not validate the `Origin` header on its development endpoints. In a Vite-hosted dev server, this exposes the developer's in-flight source files and HMR payloads.

**Practical exposure for this project.**
- Requires `npm run dev` actively running on a predictable port (default 5173).
- Requires the developer to visit an attacker-controlled website while the dev server is up.
- Requires that attacker website to know the port number.
- Worst-case payoff for the attacker: read the developer's source files while they're working.

For a single-person open-source project without sensitive business logic or credentials in the source, the leak is low-value even in the unlikely case of exploitation.

**Propagates to these packages in the tree:** `vite` (direct), `vite-node`, `vitest`, `@vitest/mocker`, `@vitest/coverage-v8`.

### 3.2 CVE — serialize-javascript RCE

| Field | Value |
|---|---|
| Advisory | [GHSA-5c6j-r48x-rmvq](https://github.com/advisories/GHSA-5c6j-r48x-rmvq) |
| CWE | CWE-96 (Improper Neutralization of Directives) |
| CVSS | 8.1 (high) |
| Vulnerable range | `serialize-javascript ≤ 7.0.4` |
| Fix | `vite-plugin-pwa 1.2.0` (bundles a newer workbox-build → newer serialize-javascript) |

**What it does.** `serialize-javascript` mishandles `RegExp.flags` and `Date.prototype.toISOString()` in a way that, when the resulting string is later executed as JavaScript (as it is in some consumer libraries), allows arbitrary code execution with the caller's privileges.

**How it reaches this project.** `vite-plugin-pwa` → `workbox-build` → `@rollup/plugin-terser` → `serialize-javascript`. Runs during `npm run build` for the PWA service-worker manifest generation.

**Practical exposure for this project.**
- Requires an attacker to inject crafted data into the serializer's input.
- Our build input is our own source code. An attacker would need to already have write access to the repo to poison the input — at which point they already have full code execution and this CVE is redundant.
- Not exploitable from an end user of the deployed app.

There is also a secondary, lower-severity advisory on the same package ([GHSA-qj8w-gfj5-8c6v](https://github.com/advisories/GHSA-qj8w-gfj5-8c6v), CVSS 5.9) — a DoS via crafted array-likes. Same exposure model: requires build-input control we don't offer.

**Propagates to these packages in the tree:** `@rollup/plugin-terser`, `workbox-build`, `vite-plugin-pwa` (direct).

### 3.3 CVE — Vite dev server path traversal

| Field | Value |
|---|---|
| Advisory | [GHSA-4w7w-66w2-5vf9](https://github.com/advisories/GHSA-4w7w-66w2-5vf9) |
| CWE | CWE-22 (Path Traversal), CWE-200 (Information Exposure) |
| CVSS | (unscored on the advisory) |
| Vulnerable range | `vite ≤ 6.4.1` |
| Fix | Vite 7+ |

**What it does.** The Vite dev server's handling of optimized-dependency source maps (`.map` files) contains a path-traversal flaw: a crafted URL can cause the server to read and return a file from outside the project root.

**Practical exposure for this project.**
- Requires the dev server to be running.
- Requires the dev server to be reachable by the attacker — i.e., bound to a non-localhost interface, or attacker on the same LAN, or developer using a tunneling tool (ngrok etc.).
- Our config binds Vite to `localhost:5173`. We never `--host` it. Exposure: zero unless a developer deliberately changes that.

**Propagates to these packages in the tree:** same as §3.1 (`vite` is the common ancestor for both esbuild and its own vulnerability).

## 4. Transitive amplification — why "10" is misleading

Raw audit output:

```
- @rollup/plugin-terser    [high]      ← effect of serialize-javascript
- @vitest/coverage-v8      [moderate]  ← effect of vitest
- @vitest/mocker           [moderate]  ← effect of vite
- esbuild                  [moderate]  ← ROOT: CVE 3.1
- serialize-javascript     [high]      ← ROOT: CVE 3.2
- vite                     [moderate]  ← ROOT: CVE 3.3 (and also effect of esbuild)
- vite-node                [moderate]  ← effect of vite
- vite-plugin-pwa          [high]      ← effect of @rollup/plugin-terser
- vitest                   [moderate]  ← effect of vite
- workbox-build            [high]      ← effect of @rollup/plugin-terser
```

Collapsing transitive effects into their root advisories:

| Root CVE | Packages flagged | Count |
|---|---|---|
| esbuild origin validation (§3.1) | esbuild, vite, vite-node, vitest, @vitest/mocker, @vitest/coverage-v8 | 6 |
| serialize-javascript RCE (§3.2) | serialize-javascript, @rollup/plugin-terser, workbox-build, vite-plugin-pwa | 4 |
| vite path traversal (§3.3) | vite (same entry also counted above; distinct advisory) | 1 (overlapping) |

npm audit does not de-duplicate by root CVE — it reports every package whose dependency tree touches a flagged version. The "10 findings" headline figure over-represents the real finding count of **3 distinct CVEs**.

## 5. Risk assessment by scenario

| Scenario | Attack path | Reachable? | Severity in our context |
|---|---|---|---|
| End user opens the deployed PWA | None of the vulnerable code is in `dist/` | ❌ | none |
| Developer running `npm run dev` on localhost | §3.1 requires adversarial cross-origin page + port guess while dev server runs | ⚠ theoretical | negligible |
| Developer running `npm run dev` exposed to LAN/internet (`--host`) | §3.1 + §3.3 both become more plausible | ⚠ real if misused | **don't do this** |
| Developer running `npm run build` | §3.2 requires attacker-controlled build input | ❌ (attacker would need repo write already) | none |
| CI pipeline running build/test | Same as above; CI builds from vetted source | ❌ | none |
| Other developer cloning and building | Same exposure as the original developer, scaled by number of devs | ⚠ theoretical | negligible |

**Bottom line: no user-facing risk; developer risk is theoretical and self-contained.**

## 6. Remediation options considered

### Option A — `npm audit fix --force` now (rejected)

What it would do: automatic major-version bumps.

| Package | Current | Audit-fix proposes |
|---|---|---|
| `vite` | `^5.4.2` | `8.0.9` |
| `vitest` | `^2.0.5` | `4.1.5` |
| `@vitest/coverage-v8` | `^2.0.5` | `4.1.5` |
| `vite-plugin-pwa` | `^0.20.5` | `1.2.0` |

**Why rejected.**
1. Three independent, simultaneous major upgrades on a scaffold we just verified green — high probability of breaking the baseline for a threat model that doesn't reach users.
2. `--force` bypasses compatibility checking. If any peer dep disagrees, the lockfile gets worse, not better.
3. The cost of fixing whatever breaks (tests, ESM/CJS interop, plugin API changes) is real; the benefit (no change to user-facing risk) is nil.

### Option B — Targeted upgrades one-at-a-time (rejected for now)

Attempt to upgrade `vite-plugin-pwa` independently to clear the high-severity chain first, then tackle Vite/Vitest later. Rejected not on its merits but on sequencing: §6.3 folds this into planned work at near-zero extra cost.

### Option C — Defer; address in the slices that already require upgrades (**selected**)

The plan already has two natural touchpoints that will cause these upgrades anyway:

- **Slice S10 (PWA wiring).** We will need to deliberately upgrade and configure `vite-plugin-pwa` to enable the service worker. Upgrading to 1.2.x at that point clears CVE §3.2's entire chain (serialize-javascript, @rollup/plugin-terser, workbox-build, vite-plugin-pwa) for free.
- **Slice S11 (pre-launch).** Re-run `npm audit`. If CVEs §3.1 and §3.3 (Vite/esbuild path) are still flagged, make a deliberate decision then: upgrade Vite for a single dedicated commit with full test pass, or ship a README note acknowledging the dev-only exposure.

This sequencing means zero wasted work and keeps risky upgrades isolated in slices where we're already exercising the upgraded code path.

## 7. Decision

**Confirmed 2026-04-22:** defer remediation. No changes to `package.json` versions as a result of this audit.
